# Introduction

## Vue d'ensemble

Canyon (signifiant "canyon", prononcé /ˈkænjən/) est une plateforme de collecte de couverture de code JavaScript. Nous résolvons les difficultés rencontrées par les développeurs et les ingénieurs QA lors de la collecte de la couverture de code des cas de test dans le cadre de tests end-to-end. Elle se compose de trois parties principales :

- Une série de [plugins](https://github.com/canyon-project/canyon/tree/main/plugins) responsables de l'adaptation à divers fournisseurs CI et de la détection des variables d'environnement.

- Un [service nodejs](https://github.com/canyon-project/canyon/tree/main/packages/canyon-backend) chargé de collecter et traiter les données de couverture et de fournir une API backend.

- Une [application frontend React](https://github.com/canyon-project/canyon/tree/main/packages/canyon-platform) responsable de l'affichage des données de couverture.

![](https://cdn.jsdelivr.net/gh/canyon-project/assets/docs/static/docs/getting-started/introduction/home-screen.png)

## Pourquoi Canyon ?

Canyon peut traiter efficacement de grandes quantités de données de couverture générées par les tests d'automatisation UI en séparant les données hit et map lors de la phase de compilation.

De plus, Canyon s'adapte aux fournisseurs CI courants, permettant d'insérer du code de sonde lors de la phase de construction et de collecter et signaler les données de couverture lors de la phase de test d'automatisation UI.

Cela permet aux équipes de développement de connaître __précisément__ et __en temps réel__ la couverture de __chaque cas de test__ d'automatisation UI actuel, afin de mieux évaluer et améliorer la qualité du code.

### Fonctionnalités

- [Précis et Efficace](/docs/core-concepts/separate-hit-and-map)：Grâce à la séparation des hit et map, les données de couverture initiales sont générées au moment de la compilation, permettant une collecte de données de couverture précise et efficace
- [SourceMap](/docs/core-concepts/restore-source-code-coverage)：Restauration précise de la couverture du code source
- [Outils de Build](/docs/installation/getting-started)：Fournit des solutions de couverture pour les outils de build courants comme next.js, vite, webpack
- [Frameworks d'Automatisation](/docs/end-to-end-testing/getting-started)：Solutions d'intégration pour les frameworks d'automatisation UI courants
- [Fournisseurs CI](/docs/reference/provider)：S'adapte aux fournisseurs CI courants (Github Actions, GitLab Runner), détecte les variables d'environnement CI


### Auto-hébergement

Si vous souhaitez avoir un contrôle total sur vos données de couverture et vos données de test, vous pouvez [auto-héberger](/docs/self-host/community-edition/prerequisites) Canyon sur vos propres infrastructures.

## Architecture

Le diagramme suivant montre l'architecture de Canyon et de certains de ses composants d'écosystème :

1. Le plugin Babel effectue l'instrumentation du code dans le pipeline CI/CD.
2. Le projet est déployé dans un environnement de test, et l'utilisateur peut choisir des tests d'automatisation UI ou des tests manuels pour déclencher les sondes de code instrumenté.
3. Les données de couverture générées sont signalées au serveur Canyon.
4. Le serveur Canyon consomme les données et génère des rapports de couverture en combinaison avec les informations de code source fournies par GitLab.
